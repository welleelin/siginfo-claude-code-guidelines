## 🎯 核心原则

### 软件工程"一头一尾"原则

> **需求阶段是"头"，测试阶段是"尾"**。两头抓牢，中间才能顺畅。

| 阶段 | 核心作用 | 关键产出 |
|------|---------|---------|
| **Phase 0: 需求文档验证** | 确定"做什么"，建立验收依据 | 需求验证报告、验收清单 |
| **Phase 1-3: 测试阶段** | 验证"做得对不对"，确保质量 | 各阶段测试报告 |

### 四大阶段原则

| 阶段 | 数据模式 | 测试重点 | 产出物 |
|------|---------|---------|--------|
| Phase 0: 需求文档验证 | - | 流程/模块/需求清晰度 | 需求验证报告 |
| Phase 1: 前端 UI 测试 | ✅ Mock | UI/交互/控制台 | 前端测试报告 |
| Phase 2: 后端 API 测试 | ❌ 真实数据 | API/数据库/边界 | API 测试报告 |
| Phase 3: 前后端联调 | ❌ 真实环境 | 全流程/闭环/业务 | 联调测试报告 |

### 测试报告原则

1. **阶段结束必写报告** - 每个阶段完成后必须生成测试报告
2. **问题描述人性化** - 用白话文描述问题，人看得懂
3. **解决方案清晰** - 记录怎么解决的，为什么这样解决
4. **文档留痕归档** - 报告写入项目文档，作为开发记录

## Phase 0: 需求文档验证

> **核心观点**：需求不清晰，测试就没有依据。必须先验证需求文档是否清晰，再进入测试阶段。

### 📋 验证范围

需求文档验证是测试工作的**前置阶段**，在 Phase 1 之前进行。

#### 核心目标

1. **确保需求清晰** - 流程清晰、模块清晰、需求清晰
2. **建立验收依据** - 需求文档是测试和验收的重要依据
3. **识别缺失点** - 发现需求文档中遗漏的内容
4. **形成需求闭环** - 帮用户梳理需求，形成完整闭环

### 问题 3：订单创建时库存扣减无事务保护

**发现时间**：2026-03-10 16:00
**问题描述**：创建订单时，如果后续步骤失败，库存已扣减但订单未创建，导致数据不一致
**问题原因**：缺少事务包装
**问题影响**：严重 - 数据一致性问题
**解决方案**：
```typescript
// 修改前
async function createOrder(userId: string, items: OrderItem[]) {
  // 扣减库存
  for (const item of items) {
    await db.products.updateOne(
      { id: item.productId },
      { $inc: { stock: -item.quantity } }
    )
  }

  // 创建订单
  return db.orders.create({ userId, items })
}

// 修改后
async function createOrder(userId: string, items: OrderItem[]) {
  const session = await db.startSession()
  session.startTransaction()

  try {
    // 检查库存
    for (const item of items) {
      const product = await db.products.findOne({ id: item.productId })
      if (product.stock < item.quantity) {
        throw new Error('库存不足')
      }
    }

    // 扣减库存
    for (const item of items) {
      await db.products.updateOne(
        { id: item.productId },
        { $inc: { stock: -item.quantity } },
        { session }
      )
    }

    // 创建订单
    const order = await db.orders.create({ userId, items }, { session })

    await session.commitTransaction()
    return order
  } catch (error) {
    await session.abortTransaction()
    throw error
  } finally {
    await session.endSession()
  }
}
```

## 二、问题分析

### 问题原因
后端获取购物车接口报错，原因是用户 Token 过期，但前端没有处理 Token 过期的情况

### 代码定位
文件：`src/api/cart.ts`
行号：第 25 行

```typescript
// 问题代码
export async function getCart() {
  const response = await fetch('/api/cart', {
    headers: {
      'Authorization': `Bearer ${token}`  // token 可能已过期
    }
  })
  return response.json()  // 没有处理错误状态
}
```

---

## 三、解决方案

### 修复方案
在前端 API 请求中添加错误处理，当 Token 过期时自动跳转到登录页

### 修复代码
```typescript
// 修复后
export async function getCart() {
  const response = await fetch('/api/cart', {
    headers: {
      'Authorization': `Bearer ${token}`
    }
  })

  if (response.status === 401) {
    // Token 过期，跳转登录页
    localStorage.removeItem('token')
    window.location.href = '/login'
    return null
  }

  if (!response.ok) {
    throw new Error('获取购物车失败')
  }

  return response.json()
}
```

## 🤖 多 Agent 测试执行模式

> **核心原则**：使用多 Agent 并行执行测试，提高测试效率和覆盖率。

### 执行模式对比

| 模式 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **并行测试模式** | 独立功能模块 | 速度快、效率高 | 需要隔离环境 |
| **流水线测试模式** | 有依赖的模块 | 顺序可控、依赖清晰 | 速度较慢 |
| **Swarm 测试模式** | 复杂业务流程 | 覆盖全面、交叉验证 | 协调复杂 |

```typescript
async function runAgentTests(agent: AgentConfig) {

    agentName: agent.name,
```

```typescript
// 流水线测试执行示例
async function runPipelineTests(config: PipelineTestConfig) {
  console.log('🚀 启动流水线测试...')

  const context = {
    startTime: new Date(),
    stages: [],
    artifacts: {},
    data: {},
    state: {},
  }

  for (const stage of config.pipeline) {
    console.log(`\n▶️ Stage ${stage.name} 开始...`)

    // 检查依赖
    if (stage.dependsOn) {
      const depsReady = checkDependencies(stage.dependsOn, context)
      if (!depsReady) {
        throw new Error(`Stage ${stage.name} 的依赖未满足`)
      }
    }

    // 执行阶段测试
    const stageResult = await executeStage(stage, context)
    context.stages.push(stageResult)

    // 传递产物
    if (config.handover.artifacts) {
      context.artifacts[stage.stage] = stageResult.artifacts
    }
    if (config.handover.data) {
      context.data[stage.stage] = stageResult.data
    }
    if (config.handover.state) {
      context.state[stage.stage] = stageResult.state
    }

    console.log(`✅ Stage ${stage.name} 完成`)

    // 如果阶段失败，停止流水线
    if (!stageResult.passed) {
      console.log(`❌ Stage ${stage.name} 失败，停止流水线`)
      break
    }
  }

  // 生成汇总报告
  await generatePipelineReport(context)

  console.log('✅ 流水线测试完成')
  return context
}
```

#### Swarm 测试执行

```typescript
// Swarm 测试执行示例
async function runSwarmTests(config: SwarmTestConfig) {
  console.log('🚀 启动 Swarm 测试...')

  // 1. 初始化 Swarm 环境
  await initializeSwarmEnvironment(config)

  // 2. 并行执行不同角色视角的测试
  const perspectives = await Promise.all(
    config.agents.map(agent => runPerspectiveTests(agent))
  )

  // 3. 交叉验证
  const validationResults = await crossValidate(
    perspectives,
    config.crossValidation
  )

  // 4. 达成共识
  const consensus = await reachConsensus(
    perspectives,
    validationResults,
    config.coordinator
  )

  // 5. 生成 Swarm 测试报告
  await generateSwarmReport({
    perspectives,
    validationResults,
    consensus,
  })

  console.log('✅ Swarm 测试完成')
  return consensus
}

// 交叉验证
async function crossValidate(
  perspectives: PerspectiveResult[],
  rules: CrossValidationConfig
) {
  const results = {
    dataConsistency: [],
    stateConsistency: [],
    behaviorConsistency: [],
  }

  // 数据一致性验证
  if (rules.dataConsistency.enabled) {
    for (const rule of rules.dataConsistency.rules) {
      const isConsistent = checkDataConsistency(perspectives, rule)
      results.dataConsistency.push({
        rule,
        passed: isConsistent,
        details: `数据 ${rule} 在所有视角中${isConsistent ? '一致' : '不一致'}`,
      })
    }
  }

  // 状态一致性验证
  if (rules.stateConsistency.enabled) {
    for (const rule of rules.stateConsistency.rules) {
      const isConsistent = checkStateConsistency(perspectives, rule)
      results.stateConsistency.push({
        rule,
        passed: isConsistent,
        details: `状态 ${rule} 在所有视角中${isConsistent ? '一致' : '不一致'}`,
      })
    }
  }

  // 行为一致性验证
  if (rules.behaviorConsistency.enabled) {
    for (const rule of rules.behaviorConsistency.rules) {
      const isConsistent = checkBehaviorConsistency(perspectives, rule)
      results.behaviorConsistency.push({
        rule,
        passed: isConsistent,
        details: `行为 ${rule} 在所有视角中${isConsistent ? '一致' : '不一致'}`,
      })
    }
  }

  return results
}

// 达成共识
async function reachConsensus(
  perspectives: PerspectiveResult[],
  validationResults: ValidationResults,
  coordinator: CoordinatorConfig
) {
  const totalTests = perspectives.reduce((sum, p) => sum + p.tests.length, 0)
  const passedTests = perspectives.reduce((sum, p) => sum + p.passed, 0)
  const consistencyRate = calculateConsistencyRate(validationResults)

  const consensus = {
    passed: consistencyRate >= coordinator.threshold && passedTests === totalTests,
    testPassRate: passedTests / totalTests,
    consistencyRate,
    threshold: coordinator.threshold,
    summary: '',
  }

  if (consensus.passed) {
    consensus.summary = `✅ Swarm 测试通过：测试通过率 ${(consensus.testPassRate * 100).toFixed(1)}%，一致性 ${(consistencyRate * 100).toFixed(1)}%`
  } else {
    consensus.summary = `❌ Swarm 测试未通过：测试通过率 ${(consensus.testPassRate * 100).toFixed(1)}%，一致性 ${(consistencyRate * 100).toFixed(1)}%（阈值 ${(coordinator.threshold * 100).toFixed(0)}%）`
  }

  return consensus
}
```

```typescript
function selectStrategy(phase: string, complexity: 'low' | 'medium' | 'high'): string {
  if (phase === 'phase-2') {
    return 'byModule'
  }

  if (phase === 'phase-3') {
    if (complexity === 'high') {
      return 'hybrid'
    }
    if (complexity === 'medium') {
      return 'byFlow'
    }
    return 'byModule'
  }

  return 'byModule'
```

## ⚠️ 重要规则

### 规则 1：需求必须闭环（Phase 0 专属）

```
用户描述需求 → AI 整合分析 → 列出功能点/页面/交互 → 用户确认
      ↑                                              ↓
      └──────────── 用户补充 ←── 发现疏漏 ←─────────┘
                              ↓
                      形成需求闭环
```

**闭环标准**：
- 正向流程完整
- 反向流程完整
- 异常流程完整
- 边界条件明确
- 验收清单明确

### 规则 2：问题自行解决

```
发现问题 → 自行分析 → 自行解决 → 验证通过 → 记录到报告 → 告知用户
```

**不要在问题未解决时就告知用户**，应该先解决完再汇报。

### 规则 3：Bug 描述人性化

```markdown
# ❌ 错误示例
问题：TypeError: Cannot read property 'id' of undefined
原因：user 为 undefined
解决：添加 user && user.id 判断

# ✅ 正确示例
问题：点击用户详情时报错
原因：用户数据还没加载完成就尝试显示，导致找不到用户信息
解决：在显示用户信息前先判断数据是否已加载，如果没加载就显示"加载中..."
```

### 规则 4：文档留痕

所有测试报告必须：
1. 写入项目 `docs/test-reports/` 目录
2. 使用 Markdown 格式
3. 包含问题和解决方案
4. 作为项目开发记录归档

### 规则 5：Mock 模式限制

| 阶段 | Mock 数据 | 真实数据 |
|------|----------|---------|
| Phase 0: 需求文档验证 | - | - |
| Phase 1: 前端 UI 测试 | ✅ 允许 | ❌ 不要求 |
| Phase 2: 后端 API 测试 | ❌ 禁止 | ✅ 必须 |
| Phase 3: 前后端联调 | ❌ 严禁 | ✅ 必须 |

### 规则 6：验收依据

Phase 0 生成的验收清单是最终验收的唯一依据：

```markdown
## 验收依据使用规则

1. Phase 0 生成的验收清单 = 最终验收标准
2. Phase 3 测试时对照验收清单逐项验证
3. 验收清单变更需要重新走需求确认流程
4. 任何功能变更都要更新验收清单
```

> **核心原则**：
> 1. **一头一尾**：需求阶段是"头"，测试阶段是"尾"，两头抓牢
> 2. **需求闭环**：帮用户梳理需求，形成完整闭环，建立验收依据
> 3. **阶段分明**：需求验证 → 前端 Mock → 后端真实 → 联调真实
> 4. **自行解决**：发现问题先解决再汇报
> 5. **报告留痕**：每个阶段必有测试报告
> 6. **人性化描述**：白话文，人看得懂
> 7. **多 Agent 协作**：并行/流水线/Swarm 模式提高效率
