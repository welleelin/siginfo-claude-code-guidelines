## 🎯 协作模式决策树

### 决策维度

基于以下 4 个维度进行决策：

| 维度 | 分类 | 说明 |
|------|------|------|
| **任务复杂度** | 简单 (<2h) / 中等 (2-8h) / 复杂 (>8h) | 任务的技术难度和工作量 |
| **依赖关系** | 强依赖 / 弱依赖 / 混合依赖 | 子任务之间的依赖程度 |
| **时间紧迫度** | 紧急 (<24h) / 正常 (1-7天) / 长期 (>7天) | 交付时间要求 |
| **资源可用性** | 单窗口 / 多窗口 / 团队 | 可用的 Agent 资源 |

### 决策流程图

```
开始
  │
  ▼
任务复杂度？
  │
  ├─ 简单 (<2h) ──────────▶ 单 Agent
  │
  ├─ 中等 (2-8h) ─────────▶ 依赖关系？
  │                          │
  │                          ├─ 强依赖 ──▶ 串行协作
  │                          │
  │                          └─ 弱依赖 ──▶ 并行协作
  │
  └─ 复杂 (>8h) ──────────▶ 时间紧迫度？
                             │
                             ├─ 紧急 (<24h) ──▶ 混合模式（最大化并行）
                             │
                             ├─ 正常 (1-7天) ──▶ 团队模式
                             │
                             └─ 长期 (>7天) ──▶ 混合模式（迭代开发）
```

### 决策矩阵

| 任务特征 | 推荐模式 | 理由 | 预期效率 |
|---------|---------|------|---------|
| 简单任务 + 单窗口 | 单 Agent | 无需协作开销 | 1x（基准） |
| 中等任务 + 强依赖 | 串行协作 | 步骤清晰，质量优先 | 1.5x |
| 中等任务 + 弱依赖 | 并行协作 | 最大化并行度 | 2-3x |
| 复杂任务 + 紧急 | 混合模式 | 平衡速度与质量 | 3-5x |
| 复杂任务 + 正常 | 团队模式 | 全面质量保障 | 2.5-4x |
| 复杂任务 + 长期 | 混合模式 | 迭代优化 | 3-5x |

## 📊 协作模式对比矩阵

### 量化对比

| 模式 | 效率提升 | 时间节省 | 质量提升 | 成本增加 | ROI |
|------|---------|---------|---------|---------|-----|
| **单 Agent** | 基准 | 0% | 基准（60%） | 基准 | 1.0x |
| **串行协作** | 1.5x | 20-30% | +40%（80%） | +30% | 1.15x |
| **并行协作** | 2-3x | 50-70% | +20%（75%） | +80% | 1.11-1.67x |
| **团队模式** | 2.5-4x | 60-75% | +60%（90%） | +100% | 1.25-2x |
| **混合模式** | 3-5x | 70-80% | +70%（92%） | +120% | 1.36-2.27x |

### 详细说明

#### 单 Agent（基准）
- **适用场景**：简单任务（<2h）、单一职责
- **优势**：无协作开销、上下文连贯
- **劣势**：效率受限、无质量保障
- **典型任务**：Bug 修复、文档更新、简单功能

#### 串行协作（1.5x 效率）
- **适用场景**：中等任务（2-8h）、强依赖关系
- **优势**：步骤清晰、质量提升 40%
- **劣势**：时间较长、并行度低
- **典型任务**：TDD 开发、代码审查、重构

**协作流程**：
```
planner → tdd-guide → code-reviewer → security-reviewer
```

#### 并行协作（2-3x 效率）
- **适用场景**：中等任务（2-8h）、弱依赖关系
- **优势**：时间节省 50-70%、高并行度
- **劣势**：协调成本高、质量提升有限
- **典型任务**：多模块开发、独立功能

**协作流程**：
```
┌─ Agent 1: 后端 API
├─ Agent 2: 前端页面
└─ Agent 3: 数据库迁移
```

#### 团队模式（2.5-4x 效率）
- **适用场景**：复杂任务（>8h）、全面质量要求
- **优势**：质量提升 60%、全流程覆盖
- **劣势**：成本高、协调复杂
- **典型任务**：大型功能、架构重构

**协作流程**：
```
architect → planner → [tdd-guide, code-reviewer, security-reviewer] → e2e-runner → doc-updater
```

#### 混合模式（3-5x 效率）
- **适用场景**：复杂任务（>8h）、灵活需求
- **优势**：最高效率、最佳质量
- **劣势**：需要精细规划
- **典型任务**：大型项目、紧急需求

**协作流程**：
```
团队模式（架构设计） → 并行协作（开发） → 团队模式（集成测试） → 串行协作（优化）
```

## 🔀 混合模式策略

### 大型项目（>1 周）

**策略**：架构设计（团队）→ 并行开发 → 集成测试（团队）→ 迭代优化

```
阶段 1：架构设计（团队模式）
├─ architect: 技术方案
├─ planner: 任务分解
└─ security-reviewer: 安全评估

阶段 2：并行开发（并行协作）
├─ Agent 1: 后端 API（tdd-guide + code-reviewer）
├─ Agent 2: 前端页面（tdd-guide + code-reviewer）
└─ Agent 3: 数据库（tdd-guide + code-reviewer）

阶段 3：集成测试（团队模式）
├─ e2e-runner: 端到端测试
├─ security-reviewer: 安全扫描
└─ build-error-resolver: 构建验证

阶段 4：迭代优化（串行协作）
└─ refactor-cleaner → code-reviewer → doc-updater
```

**效果**：
- 效率提升：4-5x
- 时间节省：75-80%
- 质量提升：+70%（92% 覆盖率）

### 紧急项目（<24h）

**策略**：最大化并行度 → 串行收尾

```
阶段 1：快速规划（15 分钟）
└─ planner: 最小可行方案

阶段 2：并行开发（并行协作）
├─ Agent 1: 核心功能（tdd-guide）
├─ Agent 2: 测试用例（tdd-guide）
└─ Agent 3: 文档（doc-updater）

阶段 3：串行收尾（串行协作）
└─ code-reviewer → e2e-runner → 部署
```

**效果**：
- 效率提升：3-4x
- 时间节省：70-75%
- 质量：75-80%（可接受）

### 维护项目

**策略**：Bug 修复（串行）+ 小功能（迭代）+ 定期维护（并行）

```
日常维护：
├─ Bug 修复：单 Agent 或串行协作
├─ 小功能：串行协作（tdd-guide → code-reviewer）
└─ 定期维护：并行协作（refactor-cleaner + doc-updater + security-reviewer）

季度重构：
└─ 团队模式（architect → planner → 并行开发 → 集成测试）
```

**效果**：
- 日常效率：1.5-2x
- 重构效率：3-4x
- 质量稳定：85-90%

## 🔧 效率优化技巧

### 瓶颈识别与优化

| 瓶颈类型 | 识别方法 | 优化方案 | 效果 |
|---------|---------|---------|------|
| **需求分析** | 规划时间 >30% | 使用 planner agent 自动分析 | 时间减少 50% |

| **人类介入** | 等待时间 >2h | 减少非必要确认点 | 时间减少 80% |

### 并行化策略

**可并行任务**：
- 后端 API + 前端页面 + 数据库迁移
- 单元测试 + 集成测试 + E2E 测试
- 代码审查 + 安全扫描 + 性能分析
- 文档生成 + 部署准备 + 监控配置

**不可并行任务**：
- 需求分析 → 架构设计（强依赖）

- 代码实现 → 代码审查（顺序依赖）
- 集成测试 → 部署（质量门禁）

## ⚠️ 反模式警告

### 1. 过度设计

**表现**：
- 简单任务使用团队模式
- 2 小时任务规划 1 小时
- 过度抽象和封装

**后果**：
- 效率降低 50%
- 成本增加 200%
- 交付延迟

**避免策略**：
- 简单任务用单 Agent
- 规划时间 <20% 总时间
- 遵循 YAGNI 原则

### 2. 过早优化

**表现**：
- 功能未完成就开始优化
- 过度关注性能细节
- 忽略核心功能

**后果**：
- 浪费时间 30-50%
- 核心功能延迟
- 可能优化错方向

**避免策略**：
- 先完成核心功能
- 基于性能测试优化
- 遵循 "Make it work, make it right, make it fast"

### 3. 错误的并行化

**表现**：
- 强依赖任务并行执行
- 并行任务过多导致协调成本高
- 忽略数据竞争

**后果**：
- 效率不升反降
- 结果不一致
- 调试困难

**避免策略**：
- 分析依赖关系
- 控制并行度（3-5 个）
- 使用确定性开发原则

### 4. 过度串行化

**表现**：
- 弱依赖任务串行执行
- 不必要的等待
- 忽略并行机会

**后果**：
- 时间浪费 50-70%
- 效率低下
- 资源闲置

**避免策略**：
- 识别弱依赖任务
- 最大化并行度
- 使用混合模式

### 5. 忽略质量门禁

**表现**：
- 跳过代码审查
- 跳过测试
- 跳过安全扫描

**后果**：
- 质量下降 40%
- 技术债务累积
- 生产事故风险

**避免策略**：
- 强制质量门禁
- 自动化检查
- 团队模式保障质量

### 6. 上下文爆炸

**表现**：
- 上下文使用率 >90%
- 频繁 compact 导致信息丢失
- 任务中断

**后果**：
- 效率降低 30%
- 信息丢失
- 需要重新开始

**避免策略**：
- 监控上下文使用率
- 80% 时自动保存
- 90% 时强制 compact

### 7. 人类介入过多

**表现**：
- 每个步骤都需要确认
- 简单决策也需要人类
- 过度依赖人类

**后果**：
- 效率降低 70%
- 自动化失效
- 人类疲劳

**避免策略**：
- 只在关键点介入
- 自动化简单决策
- 建立信任机制

### 8. 人类介入过少

**表现**：
- 架构决策无人确认
- 风险评估无人审查
- 盲目自动化

**后果**：
- 方向错误
- 风险失控
- 返工成本高

**避免策略**：
- 明确人类介入点
- 关键决策必须确认
- 风险评估必须审查

## 📈 实战案例

### 成功案例 1：电子商务网站（72 小时）

**任务**：开发完整的电子商务网站

**协作模式**：混合模式

**实施策略**：
```
阶段 1：架构设计（团队模式，4h）
├─ architect: 技术方案
├─ planner: 任务分解（156 个小任务）
└─ security-reviewer: 安全评估

阶段 2：并行开发（并行协作，48h）
├─ Agent 1: 后端 API（52 个任务）
├─ Agent 2: 前端页面（48 个任务）
├─ Agent 3: 数据库（28 个任务）
└─ Agent 4: 支付集成（28 个任务）

阶段 3：集成测试（团队模式，12h）
├─ e2e-runner: 端到端测试
├─ security-reviewer: 安全扫描
└─ build-error-resolver: 构建验证

阶段 4：优化部署（串行协作，8h）
└─ refactor-cleaner → code-reviewer → doc-updater
```

**成果**：
- 按时完成（72h）
- 测试覆盖率：92%
- 无重大返工
- 客户满意度：高

**关键成功因素**：
1. 任务分解细致（156 个小任务）
2. 每小时自动保存进度
3. 每天生成详细进度报告
4. 关键决策点有人类确认
5. 上下文达到 80% 时自动 compact

### 成功案例 2：认证系统重构（2.5 天）

**任务**：重构整个认证系统

**协作模式**：团队模式

**实施策略**：
```
阶段 1：规划（团队模式，4h）
├─ architect: 新架构设计
├─ planner: 任务分解（47 个子任务）
└─ security-reviewer: 安全评估

阶段 2：开发（串行协作，40h）
└─ tdd-guide → code-reviewer → security-reviewer（每个子任务）

阶段 3：测试（团队模式，12h）
├─ e2e-runner: 端到端测试
└─ security-reviewer: 安全扫描

阶段 4：部署（串行协作，4h）
└─ build-error-resolver → doc-updater
```

**成果**：
- 实际耗时：2.5 天（提前完成）
- 测试覆盖率：从 60% 提升到 92%
- 无重大返工
- 安全性提升 80%

**关键成功因素**：
1. 分解为 47 个子任务
2. 每个子任务独立执行和验证
3. 每 5 分钟自动保存状态
4. 每天生成详细进度报告
5. 关键决策点有人类确认

### 成功案例 3：紧急 Bug 修复（4 小时）

**任务**：修复生产环境严重 Bug

- 4 小时内修复并部署

- 回归测试覆盖率：100%

**关键成功因素**：
1. 快速规划（15 分钟）
2. 最大化并行度（3 个 Agent）
3. 准备回滚方案
4. 严格验证流程

### 失败案例 1：认证模块重构（预计 8h，实际 24h）

**任务**：重构用户认证模块

**协作模式**：单 Agent（错误选择）

**失败原因**：
1. 任务分解不够细，试图一次性重构多个组件
2. 没有定期保存状态，中途崩溃后丢失进度
3. 上下文达到 90% 才注意到，compact 后丢失关键信息
4. 遇到错误时重复尝试相同操作，浪费 4 小时
5. 没有通知人类介入，在死胡同里越走越远

**后果**：
- 时间超支 200%
- 质量下降（测试覆盖率仅 65%）
- 技术债务累积
- 团队士气受挫

**改进措施**：
- 使用团队模式
- 强制任务分解
- 强制状态保存
- 建立检查点机制
- 实现错误分级处理
- 明确人类介入点

### 失败案例 2：报表导出功能（预计 8h，实际 3 天）

**任务**：实现报表导出功能

**协作模式**：并行协作（错误选择，任务有强依赖）

**失败原因**：
1. 忽略了任务之间的强依赖关系
2. 并行开发导致接口不一致
3. 集成时发现大量冲突
4. 返工成本高

**后果**：
- 时间超支 800%
- 大量返工
- 代码质量差
- 客户不满

**改进措施**：
- 分析依赖关系
- 使用串行协作或混合模式
- 先定义接口，再并行开发
- 增加集成测试

### 失败案例 3：过度并行化（预计 2 天，实际 5 天）

**任务**：开发多个独立功能

**协作模式**：并行协作（过度并行，10 个 Agent）

**失败原因**：
1. 并行任务过多（10 个 Agent）
2. 协调成本高于并行收益
3. 上下文管理困难
4. 人类无法及时响应所有确认请求

**后果**：
- 时间超支 150%
- 协调成本高
- 人类疲劳
- 效率不升反降

**改进措施**：
- 控制并行度（3-5 个 Agent）
- 使用混合模式
- 分批并行执行
- 减少人类介入点
