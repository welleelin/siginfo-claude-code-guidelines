# 长期运行 Agent 最佳实践

> 基于 Anthropic 工程团队经验和社区实践总结

---

## 📋 概述

长期运行的 AI Agent（长时间、多步骤任务）面临独特挑战：

- **上下文管理** - Token 使用量增长导致上下文超限
- **状态保持** - 任务中断后如何恢复
- **错误恢复** - 遇到错误时如何优雅处理
- **人类介入** - 何时需要人类确认和决策
- **进度追踪** - 如何知道任务执行到哪里

本文档总结了应对这些挑战的最佳实践。

---

## 🎯 核心原则

### 1. 明确的任务边界

**问题**：Agent 试图一次性完成太多任务，导致上下文爆炸。

**解决方案**：
- 将大任务分解为独立的小任务
- 每个任务有明确的输入、输出和完成标准
- 任务之间通过文件或 API 传递结果

**示例**：
```
❌ 坏任务："开发完整的用户管理系统"
✅ 好任务 1："创建用户数据表结构"
✅ 好任务 2："实现用户注册 API"
✅ 好任务 3："实现用户登录 API"
✅ 好任务 4："创建用户管理页面"
```

### 2. 强制的上下文管理

**问题**：随着任务执行，上下文使用量增长，最终超限。

**解决方案**：
- **实时监控**：每 30 秒检查上下文使用率
- **阈值预警**：70% 预警，80% 准备压缩，90% 强制压缩
- **自动保存**：达到 80% 时自动保存关键信息到 Memory
- **自动压缩**：达到 90% 时自动执行 compact

**流程**：
```
检测使用率 → 判断阈值 → 触发动作
    │
    ├── 70% → 发送 P2 通知提醒
    ├── 80% → 自动保存到 Memory
    └── 90% → 自动 compact + 恢复状态
```

### 3. 可恢复的状态管理

**问题**：任务中断后无法恢复，需要重新开始。

**解决方案**：
- **状态快照**：定期保存任务状态到文件
- **检查点**：每个关键步骤后创建检查点
- **恢复机制**：从检查点读取状态继续执行

**状态快照内容**：
```javascript
const stateSnapshot = {
  taskId: 'task-52',
  phase: 'implementation',
  currentStep: '编写登录页面组件',
  completedSteps: ['需求分析', '任务规划', '测试先行'],
  variables: { ... },
  decisions: [
    { timestamp: '...', decision: '...', reason: '...' }
  ],
  filesModified: ['src/views/login/index.vue'],
  testResults: { passed: 3, failed: 0 }
}
```

### 4. 分层错误处理

**问题**：遇到错误时 Agent 不知所措或重复尝试相同操作。

**解决方案**：
- **错误分级**：
  - L1（轻微）：自动重试（如网络超时）
  - L2（中等）：尝试备选方案（如此 API 不可用换另一个）
  - L3（严重）：通知人类介入（如权限问题、数据丢失）

- **重试策略**：
  - 指数退避：1 分钟、2 分钟、4 分钟、8 分钟
  - 最大重试次数：3 次
  - 记录每次错误和重试

**示例**：
```
错误：API 请求超时
├─ 重试 1（等待 1 分钟）→ 失败
├─ 重试 2（等待 2 分钟）→ 失败
├─ 重试 3（等待 4 分钟）→ 失败
└─ 升级：发送 P1 通知，请求人类介入
```

### 5. 明确的人类介入点

**问题**：Agent 在需要人类决策时继续执行，导致方向错误。

**解决方案**：
- **预定义介入点**：明确哪些决策需要人类确认
- **阻塞式等待**：在获得确认前暂停执行
- **多渠道通知**：通过飞书/钉钉/短信等通知人类

**需要人类确认的场景**：
- 任务规划完成后
- 架构决策点
- 风险评估和接受
- 测试失败是否继续
- 数据删除操作
- 发布决策

### 6. 详细的进度追踪

**问题**：不知道任务执行到哪里，无法向人类汇报。

**解决方案**：
- **进度日志**：每个步骤完成后记录
- **状态看板**：可视化展示任务进度
- **定时报告**：每小时/每天生成进度报告

**进度日志格式**：
```markdown
## [2026-03-03 10:00] - 任务 52：实现用户登录

### 已完成步骤
- [x] 需求分析 (10 分钟)
- [x] 任务规划 (5 分钟)
- [x] 编写测试用例 (15 分钟)
- [ ] 实现功能 (进行中 - 50%)

### 当前活动
正在编写登录页面组件，已完成表单验证逻辑

### 下一步
- 实现 API 调用逻辑
- 处理登录响应
- 保存 Token

### 遇到的问题
无
```

## 📊 任务分解模式

### 模式 1：阶段式分解

将任务按阶段分解，每个阶段独立执行：

```
任务：开发用户管理系统

阶段 1：需求分析
├─ 输出：需求文档
└─ 检查点：需求评审通过

阶段 2：设计
├─ 输出：技术方案、数据库设计
└─ 检查点：设计评审通过

阶段 3：实现
├─ 输出：可运行代码
└─ 检查点：单元测试通过

阶段 4：测试
├─ 输出：测试报告
└─ 检查点：E2E 测试通过

阶段 5：部署
├─ 输出：线上服务
└─ 检查点：健康检查通过
```

### 模式 2：并行分解

将独立的任务并行执行：

```
任务：开发完整功能

Agent 1：后端 API
├─ 创建 Controller
├─ 实现 Service
└─ 编写单元测试

Agent 2：前端页面
├─ 创建页面组件
├─ 实现 UI 逻辑
└─ 编写 E2E 测试

Agent 3：数据库
├─ 创建表结构
├─ 编写迁移脚本
└─ 准备测试数据
```

### 模式 3：迭代式分解

通过多轮迭代逐步完善：

```
迭代 1：MVP（最小可行产品）
└─ 核心功能可用

迭代 2：增强
├─ 添加错误处理
└─ 改进用户体验

迭代 3：完善
├─ 添加日志
├─ 添加监控
└─ 性能优化
```

## 🔧 实用工具和命令

### 上下文管理工具

```bash
# 查看当前上下文使用量
/context usage

# 手动触发 compact
/context compact

# 查看上下文历史
/context history --hours 24

# 配置阈值
/context config set --warning 70 --compact 90
```

### 状态管理工具

```bash
# 保存状态快照
/state save --reason "checkpoint"

# 恢复状态
/state restore --snapshot latest

# 查看历史快照
/state list
```

### 任务管理工具

```bash
# 查看当前任务
/task current

# 查看任务进度
/task progress

# 更新任务状态
/task status --state blocked --reason "等待 API"
```

## 🎓 案例研究

### 案例 1：成功的大型任务

**任务**：开发完整的电子商务网站（耗时 72 小时）

**成功因素**：
1. 任务分解为 156 个小任务
2. 每个小任务独立执行和验证
3. 每小时自动保存进度到 Memory
4. 每天生成详细进度报告
5. 关键决策点（技术选型、架构设计）有人类确认
6. 上下文达到 80% 时自动 compact，持续执行不中断

**结果**：
- 按时完成
- 测试覆盖率 92%
- 无重大返工

### 案例 2：失败的中型任务

**任务**：重构用户认证模块（预计 8 小时，实际耗时 24 小时）

**失败原因**：
1. 任务分解不够细，试图一次性重构多个组件
2. 没有定期保存状态，中途崩溃后丢失进度
3. 上下文达到 90% 才注意到，compact 后丢失关键信息
4. 遇到错误时重复尝试相同操作，浪费 4 小时
5. 没有通知人类介入，在死胡同里越走越远

**教训**：
- 必须强制任务分解
- 必须定期保存状态
- 必须设置上下文预警
- 必须有错误分级处理
- 必须明确人类介入点
