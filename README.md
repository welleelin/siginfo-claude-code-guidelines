# sig-claude-code-guidelines

> 一套经过实战验证的 AI 辅助软件开发规范，让团队开发像流水线一样高效

---

## 📖 这是什么？

这是一套**完整的行动准则**，告诉你在用 Claude Code 开发项目时：

- **开发前**要做什么准备
- **开发中**遵循什么流程
- **完成后**如何验证质量

就像工厂的流水线作业指导书，每个步骤都有明确的规范，确保产出质量稳定可靠。

---

## 🎯 为什么要做这个？

### 问题场景

你是否遇到过：

1. **开发前没想清楚**，写到一半发现方向错了，推倒重来
2. **测试后写**，为了赶进度跳过测试，上线后 bug 频发
3. **代码提交混乱**，不知道谁改了什么，出了问题找不到原因
4. **新人上手慢**，没有文档指导，全靠口口相传
5. **AI 使用随意**，没有发挥 AI 的真正能力，效率低下

### 我们的解决方案

这套行动准则就是为了解决这些问题：

```
✅ 明确的开发流程 - 不再迷茫下一步该做什么
✅ 测试驱动开发 - 先写测试，保证质量
✅ 完整的文档记录 - 任何任务都可追溯
✅ 新人友好 - 按文档操作，快速上手
✅ AI 能力最大化 - 充分利用插件，效率翻倍
```

## 🔄 完整开发流程（文字版）

### 第一阶段：会话启动准备

**目标**：确保环境和上下文准备就绪

**步骤**：

1. **检查插件环境** - 确认必备插件已安装
   - bmad-method（需求分析）
   - everything-claude-code（命令库）
   - workflow-studio（流程图）
   - pencil（UI 设计）

2. **读取项目文档** - 按顺序阅读
   - `CLAUDE.md` - 项目概述
   - `ACTION_GUIDELINES.md` - 行动准则
   - `MEMORY.md` - 项目当前状态
   - `task.json` - 下一个待执行任务

3. **环境检查** - 运行命令
   ```bash
   pwd                # 确认目录
   git status         # 检查更改
   git log -5         # 查看历史
   ```

**产出**：清晰的项目上下文，明确下一个任务

---

### 第二阶段：任务规划（含流程图可视化）

**目标**：在写代码前，先想清楚要做什么，并用流程图可视化呈现

**步骤**：

1. **运行规划命令** - `/plan "任务描述"`

2. **判断是否需要流程图**

   以下情况**需要**生成流程图：
   - ✅ 涉及多个步骤的任务
   - ✅ 有分支判断的业务逻辑
   - ✅ 需要多人协作的复杂任务
   - ✅ 有时序关系的 API 调用
   - ✅ 状态流转复杂的功能

3. **自动生成流程图** - 使用 workflow-studio

   ```markdown
   ## 业务流程图

   ```flow
   TD
   A[用户打开登录页] --> B{是否已登录}
   B -->|是 | C[跳转到首页]
   B -->|否 | D[显示登录表单]
   D --> E[输入账号密码]
   E --> F[点击登录按钮]
   F --> G{验证结果}
   G -->|成功 | H[跳转首页]
   G -->|失败 | I[显示错误提示]
   I --> D
   ```
   ```

4. **确认或修改计划**
   - 同意：回复 "yes" 或 "proceed"
   - 修改：回复 "modify: [你的修改]"

**产出**：清晰的实现计划 + 可视化流程图

---

### 第三阶段：TDD 开发（核心）

**目标**：用测试驱动的方式高质量实现功能

#### 步骤 1：RED（写失败测试）

```
1. 根据需求编写测试用例
2. 运行测试，确认失败（红色）
3. 记录测试文件位置
```

**为什么先写失败测试？**
- 确保你清楚需求是什么
- 避免过度开发
- 测试覆盖率自然达标

#### 步骤 2：GREEN（实现功能）

```
1. 编写最小代码让测试通过
2. 不要优化，先实现功能
3. 运行测试，看到绿色通过
```

**为什么只写最小代码？**
- 避免过度设计
- 快速验证方向正确
- 后续重构更有针对性

#### 步骤 3：REFACTOR（重构优化）

```
1. 优化代码结构
2. 消除重复
3. 改进命名
4. 确保测试仍然通过
```

**为什么最后重构？**
- 功能正确后再优化
- 有测试保护，重构放心
- 避免过早优化

#### 步骤 4：VERIFY（验证闭环）

```
1. 运行全部相关测试
2. 生成测试报告
3. 分析失败原因
4. 修复或记录问题
```

**产出**：通过测试的功能代码，测试报告

---

### 第四阶段：验证与提交

**目标**：确保质量达标后再提交

**检查清单**：

```
□ 构建成功（npm run build / mvn package）
□ 测试通过（所有相关测试用例）
□ 失败修复（或记录原因）
□ 代码审查（无严重问题）
□ 文档更新（MEMORY.md、task.json）
□ Git 提交（规范格式）
```

**Git 提交格式**：
```
feat: 实现用户登录功能 (任务 5)

- 创建登录页面 views/login/index.vue
- 实现认证 API api/auth.ts
- 添加 E2E 测试 e2e/05-login.spec.ts
- 测试覆盖：3 个新测试用例
- 需求关联：FR-001
```

**产出**：高质量的代码提交

## 🧪 E2E 测试验证流程（分层进行）

### 为什么分层测试？

想象一下：

- 前端页面有问题，但不知道是前端 bug 还是后端 bug
- 前后端一起测，失败了很难定位问题
- 分层测试，每层独立验证，问题无处遁形

### 三层测试流程

```
第一层：前端 Mock 测试
├── 目的：验证前端页面交互正确
├── 方法：用 Mock 数据，不依赖后端
├── 通过标准：所有页面测试通过
└── 完成后：进入第二层

第二层：后端 TDD 测试
├── 目的：验证后端 API 功能正确
├── 方法：单元测试 + 集成测试
├── 通过标准：单元测试 80%+ 覆盖
└── 完成后：进入第三层

第三层：前后端联调测试
├── 目的：验证前后端集成正常
├── 方法：E2E 测试，真实 API
├── 通过标准：所有 E2E 测试通过
└── 完成后：可以提交代码
```

## 🤖 多 Agent 协作模式

### 什么是 Agent？

Agent 是 Claude Code 中的"专家角色"，每个 Agent 擅长不同领域：

| Agent | 擅长领域 | 何时使用 |
|-------|---------|---------|
| Planner | 任务规划 | 开始新任务前 |
| Architect | 系统设计 | 架构设计时 |
| TDD-Guide | 测试驱动开发 | 写代码时 |
| Code-Reviewer | 代码审查 | 完成后 |
| Build-Resolver | 构建修复 | 编译失败时 |
| E2E-Runner | E2E 测试 | 运行测试时 |

### 任务分配策略

**简单任务（单 Agent）**
- 修改一个文件
- 修复小 bug
- 添加简单功能

**中等任务（并行 Agent）**
- 开发一个模块
- API + 页面同时开发

**复杂任务（Team 模式）**
- 新功能从 0 到 1
- 架构变更

## 📊 质量门禁

### 什么是质量门禁？

质量门禁是任务完成的"及格线"，不达标不能进入下一任务。

## 🔔 通知与确认机制

### 自动化与人类介入边界

| 任务类型 | 执行方式 | 说明 |
|---------|---------|------|
| ✅ 环境检查 | 自动执行 | 插件检查、git 状态 |
| ✅ 代码检查 | 自动执行 | lint、build、test |
| ⚠️ 任务规划 | 人类确认 | 确认可行性后执行 |
| ⚠️ 代码审查 | 人类确认 | 确认修复方案 |
| 🔴 需求确认 | 必须人类 | 业务理解和优先级 |
| 🔴 架构决策 | 必须人类 | 技术选型和方向 |

### 通知触发条件

| 事件 | 级别 | 通知渠道 | 响应时间 |
|------|------|---------|---------|
| 规划完成待确认 | P2 | 飞书 + 站内信 | 2 小时 |
| 测试失败待决策 | P2 | 飞书 + 站内信 | 2 小时 |
| 遇到阻塞需介入 | P1 | 飞书 + 邮件 | 30 分钟 |
| 架构决策点 | P1 | 飞书 + 邮件 | 30 分钟 |
| 任务完成 | P3 | 站内信 | 24 小时 |
| 紧急错误 | P0 | 飞书 + 电话 | 立即 |

### 确认反馈机制

每条通知都带有确认按钮，用户确认后会：

- ✅ 记录确认凭证
- ✅ 停止后续通知
- ✅ 触发后续流程
- ✅ 生成确认日志

**防止重复通知**：一旦确认，系统不会再发送相同通知。

## 🤖 与 Claude Monitor UI 集成

本项目定义的行动准则需要与 `claude-monitor-ui` 项目配合使用：

| 功能 | 负责项目 | 说明 |
|------|---------|------|
| 行动准则定义 | sig-claude-code-guidelines | 定义流程、规范、模板 |
| 任务监督 | claude-monitor-ui | PM Agent 自动监督各窗口任务状态 |
| 通知发送 | claude-monitor-ui | 多渠道通知（飞书/钉钉/短信等） |
| 确认反馈 | claude-monitor-ui | 接收用户确认，生成确认凭证 |

### 对接接口

行动准则项目需要在任务中调用以下 Claude Monitor UI 的 API：

```javascript
// 1. 发送通知
POST http://localhost:8083/api/notification/send
{
  "level": "P2",
  "type": "plan_confirm",
  "taskId": "task-52",
  "title": "任务规划待确认",
  "content": "...",
  "channels": ["feishu", "in-app"],
  "timeout": 7200000
}

// 2. 确认反馈
POST http://localhost:8083/api/notification/confirm
{
  "notifyId": "notify-xxx",
  "userId": "user-001",
  "action": "confirm",
  "taskId": "task-52"
}

// 3. 任务状态更新
POST http://localhost:8083/api/task/status
{
  "taskId": "task-52",
  "status": "waiting_human|in_progress|blocked|completed",
  "notifyLevel": "P2"
}

// 4. 获取任务列表
GET http://localhost:8083/api/tasks?status=pending
```
