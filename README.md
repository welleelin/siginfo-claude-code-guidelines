# sig-claude-code-guidelines

> 一套经过实战验证的 AI 辅助软件开发规范，让团队开发像流水线一样高效

---

## 📖 这是什么？

这是一套**完整的行动准则**，告诉你在用 Claude Code 开发项目时：

- **开发前**要做什么准备
- **开发中**遵循什么流程
- **完成后**如何验证质量

就像工厂的流水线作业指导书，每个步骤都有明确的规范，确保产出质量稳定可靠。

---

## 🎯 为什么要做这个？

### 问题场景

你是否遇到过：

1. **开发前没想清楚**，写到一半发现方向错了，推倒重来
2. **测试后写**，为了赶进度跳过测试，上线后 bug 频发
3. **代码提交混乱**，不知道谁改了什么，出了问题找不到原因
4. **新人上手慢**，没有文档指导，全靠口口相传
5. **AI 使用随意**，没有发挥 AI 的真正能力，效率低下

### 我们的解决方案

这套行动准则就是为了解决这些问题：

```
✅ 明确的开发流程 - 不再迷茫下一步该做什么
✅ 测试驱动开发 - 先写测试，保证质量
✅ 完整的文档记录 - 任何任务都可追溯
✅ 新人友好 - 按文档操作，快速上手
✅ AI 能力最大化 - 充分利用插件，效率翻倍
```

### 第三阶段：TDD 开发（核心）

**目标**：用测试驱动的方式高质量实现功能

#### 步骤 1：RED（写失败测试）

```
1. 根据需求编写测试用例
2. 运行测试，确认失败（红色）
3. 记录测试文件位置
```

**为什么先写失败测试？**
- 确保你清楚需求是什么
- 避免过度开发
- 测试覆盖率自然达标

#### 步骤 2：GREEN（实现功能）

```
1. 编写最小代码让测试通过
2. 不要优化，先实现功能
3. 运行测试，看到绿色通过
```

**为什么只写最小代码？**
- 避免过度设计
- 快速验证方向正确
- 后续重构更有针对性

#### 步骤 3：REFACTOR（重构优化）

```
1. 优化代码结构
2. 消除重复
3. 改进命名
4. 确保测试仍然通过
```

**为什么最后重构？**
- 功能正确后再优化
- 有测试保护，重构放心
- 避免过早优化

#### 步骤 4：VERIFY（验证闭环）

```
1. 运行全部相关测试
2. 生成测试报告
3. 分析失败原因
4. 修复或记录问题
```

**产出**：通过测试的功能代码，测试报告

## 🧪 E2E 测试验证流程（分层进行）

### 为什么分层测试？

想象一下：

- 前端页面有问题，但不知道是前端 bug 还是后端 bug
- 前后端一起测，失败了很难定位问题
- 分层测试，每层独立验证，问题无处遁形

### 三层测试流程

```
第一层：前端 Mock 测试
├── 目的：验证前端页面交互正确
├── 方法：用 Mock 数据，不依赖后端
├── 通过标准：所有页面测试通过
└── 完成后：进入第二层

第二层：后端 TDD 测试
├── 目的：验证后端 API 功能正确
├── 方法：单元测试 + 集成测试
├── 通过标准：单元测试 80%+ 覆盖
└── 完成后：进入第三层

第三层：前后端联调测试
├── 目的：验证前后端集成正常
├── 方法：E2E 测试，真实 API
├── 通过标准：所有 E2E 测试通过
└── 完成后：可以提交代码
```

## ⚠️ Mock 模式使用规范

### 核心原则

> **前端 Mock 测试通过后，后端 API 测试和前后端结合测试必须关闭 Mock，使用真实数据进行测试。**

### Mock 模式使用场景

| 阶段 | 是否使用 Mock | 说明 |
|------|-------------|------|
| **前端开发阶段** | ✅ 使用 | 前端独立开发时，使用 Mock 数据验证 UI 交互 |
| **前端 Mock 测试** | ✅ 使用 | 验证前端组件逻辑，不依赖后端 |
| **后端 API 测试** | ❌ **禁止** | 必须用真实数据测试后端接口 |
| **前后端联调测试** | ❌ **禁止** | 必须连接真实后端，使用真实数据 |
| **E2E 端到端测试** | ❌ **禁止** | 完整流程必须使用真实 API 和数据 |

### 特殊情况处理

**场景**：为了测试完整流程，但某些后端接口尚未开发完成

**处理方式**：
1. 在**未开发的接口**上可以使用 Mock
2. 已开发的接口必须使用真实 API
3. **必须做好标记**，便于后续替换

**标记示例**：

```javascript
// TODO: Mock 替换 - 用户认证接口待开发后移除此 Mock
// 关联任务：TASK-15
const loginApi = mockLoginApi  // ⚠️ TEMPORARY MOCK

// ✅ 已实现 - 使用真实 API
const getUserProfile = realGetUserProfile
```

**Mock 标记规范**：
- 使用 `// ⚠️ MOCK:` 或 `// TODO: Mock 替换` 注释
- 注明关联的任务 ID 或接口名称
- 在 `PENDING_TESTS.md` 中记录待替换的 Mock 清单

### 违反规范的后果

```
❌ 错误做法：前后端联调时仍使用 Mock
   → 结果：上线后接口调用失败，线上事故

✅ 正确做法：联调前关闭所有 Mock
   → 结果：问题在测试阶段暴露，安全上线
```

### Mock 检查清单

在前端开发完成、准备进入前后端联调前，必须执行：

```
□ 1. 扫描代码中的 Mock 标记
   grep -r "⚠️ MOCK" ./src
   grep -r "TODO: Mock 替换" ./src

□ 2. 确认已实现接口的 Mock 已移除
   - 已开发接口：关闭 Mock，使用真实 API
   - 未开发接口：保留 Mock，做好标记

□ 3. 更新 PENDING_TESTS.md
   - 记录仍在使用 Mock 的接口
   - 注明预计替换时间

□ 4. 配置测试环境
   - 前端测试环境：关闭 Mock 开关
   - E2E 测试环境：确认 API_BASE_URL 指向真实后端
```

### 问题背景

- ❌ 重复使用 Mock 模式进行测试

## 🔌 端口冲突排查规范

### 核心原则

> **执行任何测试前，必须先排查端口占用情况。如果发现端口被占用，必须避开已占用的端口，严禁强制占用或终止正在使用的进程。**

### 端口排查流程

在启动测试前，自动执行以下检查：

```bash
# Step 1: 识别测试需要使用的端口
# 常见端口：
# - 前端开发服务器：3000, 5173, 8080
# - 后端 API 服务器：8000, 8080, 3333
# - 数据库：5432 (PostgreSQL), 3306 (MySQL), 27017 (MongoDB)
# - E2E 测试：49342 (Playwright), 9515 (ChromeDriver)

# Step 2: 检查端口占用状态
# macOS / Linux:
lsof -i :<端口号>
# 或
netstat -an | grep <端口号>

# Windows:
netstat -ano | findstr :<端口号>

# Step 3: 判断占用进程
- 如果是测试相关的必要服务 → 记录，尝试复用
- 如果是无关进程 → 更换测试端口，不终止该进程
- 如果是残留测试进程 → 清理后重启
```

### 端口冲突处理策略

| 场景 | 处理方式 | 示例命令 |
|------|---------|---------|
| **端口空闲** | 直接使用 | 启动服务 |
| **被必要服务占用** | 复用现有服务 | 不重启，直接使用 |
| **被无关进程占用** | 更换测试端口 | `PORT=3001 npm run dev` |
| **被残留测试进程占用** | 清理后重启 | `kill <PID>` 后重启 |

### 端口检查清单

每次执行测试前自动检查：

```
□ 1. 识别测试所需端口
   - 前端测试端口：____
   - 后端 API 端口：____
   - 数据库端口：____
   - 其他服务端口：____

□ 2. 检查端口占用状态
   lsof -i :<端口>  # 每个端口检查

□ 3. 处理冲突
   - 端口空闲 → ✅ 直接使用
   - 端口被占用 → ✅ 更换其他端口（如 3000→3001）
   - ❌ 禁止：强制终止正在使用的进程

□ 4. 记录端口配置
   - 更新 .env.test 文件
   - 通知测试相关服务
```

### 自动端口检测脚本

在测试脚本中加入自动检测逻辑：

```bash
#!/bin/bash
# check-port.sh - 端口占用检查脚本

check_port() {
  local PORT=$1
  if lsof -Pi :$PORT -sTCP:LISTEN -t >/dev/null ; then
    echo "⚠️  端口 $PORT 已被占用"
    echo "   占用进程：$(lsof -i :$PORT | grep LISTEN | awk '{print $1, $2}')"
    return 1
  else
    echo "✅ 端口 $PORT 可用"
    return 0
  fi
}

# 使用示例
# check_port 3000 || export PORT=3001
```

### 端口配置示例

在 `.env.test` 或 `playwright.config.ts` 中配置：

```bash

# 前端测试服务器

# 后端 API 服务器

```

```typescript

import { defineConfig } from '@playwright/test'

export default defineConfig({
  webServer: {
    command: 'npm run dev',
    // 动态端口：优先使用配置端口，冲突时自动递增
    port: process.env.FRONTEND_TEST_PORT ?
          parseInt(process.env.FRONTEND_TEST_PORT) : 3000,
    reuseExistingServer: true,  // ✅ 复用已存在的服务器
    timeout: 120 * 1000,
  },
})
```

### 严禁行为

```
❌ 错误做法 1: 发现端口占用，直接 kill 占用进程
   → 风险：可能终止生产服务或其他重要进程

❌ 错误做法 2: 强制绑定端口，忽略冲突
   → 风险：测试启动失败或行为异常

❌ 错误做法 3: 不检查端口，直接启动测试
   → 风险：端口冲突导致测试失败，浪费时间排查

✅ 正确做法：先检查，再决定复用或更换端口
```

### 端口冲突应急处理

如果测试过程中遇到端口冲突错误：

```bash
# 1. 立即停止测试
Ctrl+C

# 2. 检查端口占用
lsof -i :3000

# 3. 根据情况处理
# 情况 A: 被其他测试占用 → 等待或更换端口
# 情况 B: 被残留进程占用 → kill 后重试
# 情况 C: 被生产服务占用 → 更换测试端口

# 4. 更新配置，重新运行测试
export FRONTEND_TEST_PORT=3001
npm run test:e2e
```

## 🤖 多 Agent 协作模式

### 什么是 Agent？

Agent 是 Claude Code 中的"专家角色"，每个 Agent 擅长不同领域：

| Agent | 擅长领域 | 何时使用 |
|-------|---------|---------|
| Planner | 任务规划 | 开始新任务前 |
| Architect | 系统设计 | 架构设计时 |
| TDD-Guide | 测试驱动开发 | 写代码时 |
| Code-Reviewer | 代码审查 | 完成后 |
| Build-Resolver | 构建修复 | 编译失败时 |
| E2E-Runner | E2E 测试 | 运行测试时 |

### 任务分配策略

**简单任务（单 Agent）**
- 修改一个文件
- 修复小 bug
- 添加简单功能

**中等任务（并行 Agent）**
- 开发一个模块
- API + 页面同时开发

**复杂任务（Team 模式）**
- 新功能从 0 到 1
- 架构变更

## 🔔 通知与确认机制

### 自动化与人类介入边界

| 任务类型 | 执行方式 | 说明 |
|---------|---------|------|
| ✅ 环境检查 | 自动执行 | 插件检查、git 状态 |
| ✅ 代码检查 | 自动执行 | lint、build、test |
| ⚠️ 任务规划 | 人类确认 | 确认可行性后执行 |
| ⚠️ 代码审查 | 人类确认 | 确认修复方案 |
| 🔴 需求确认 | 必须人类 | 业务理解和优先级 |
| 🔴 架构决策 | 必须人类 | 技术选型和方向 |

### 通知触发条件

| 事件 | 级别 | 通知渠道 | 响应时间 |
|------|------|---------|---------|
| 规划完成待确认 | P2 | 飞书 + 站内信 | 2 小时 |
| 测试失败待决策 | P2 | 飞书 + 站内信 | 2 小时 |
| 遇到阻塞需介入 | P1 | 飞书 + 邮件 | 30 分钟 |
| 架构决策点 | P1 | 飞书 + 邮件 | 30 分钟 |
| 任务完成 | P3 | 站内信 | 24 小时 |
| 紧急错误 | P0 | 飞书 + 电话 | 立即 |

### 确认反馈机制

每条通知都带有确认按钮，用户确认后会：

- ✅ 记录确认凭证
- ✅ 停止后续通知
- ✅ 触发后续流程
- ✅ 生成确认日志

**防止重复通知**：一旦确认，系统不会再发送相同通知。

## 🤖 与 Claude Monitor UI 集成

本项目定义的行动准则需要与 `claude-monitor-ui` 项目配合使用：

| 功能 | 负责项目 | 说明 |
|------|---------|------|
| 行动准则定义 | sig-claude-code-guidelines | 定义流程、规范、模板 |
| 任务监督 | claude-monitor-ui | PM Agent 自动监督各窗口任务状态 |
| 通知发送 | claude-monitor-ui | 多渠道通知（飞书/钉钉/短信等） |
| 确认反馈 | claude-monitor-ui | 接收用户确认，生成确认凭证 |

### 对接接口

行动准则项目需要在任务中调用以下 Claude Monitor UI 的 API：

```javascript
// 1. 发送通知
POST http://localhost:8083/api/notification/send
{
  "level": "P2",
  "type": "plan_confirm",
  "taskId": "task-52",
  "title": "任务规划待确认",
  "content": "...",
  "channels": ["feishu", "in-app"],
  "timeout": 7200000
}

// 2. 确认反馈
POST http://localhost:8083/api/notification/confirm
{
  "notifyId": "notify-xxx",
  "userId": "user-001",
  "action": "confirm",
  "taskId": "task-52"
}

// 3. 任务状态更新
POST http://localhost:8083/api/task/status
{
  "taskId": "task-52",
  "status": "waiting_human|in_progress|blocked|completed",
  "notifyLevel": "P2"
}

// 4. 获取任务列表
GET http://localhost:8083/api/tasks?status=pending
```
